RUST BASICS

Learn the fundamental structure of Rust programs

Module 1: Rust Program Structure

Rust is a systems programming language that runs blazingly fast, prevents segfaults, and guarantees thread safety. Think of Rust as the perfect blend of C++'s performance with Python's safety and developer experience.

Every Rust program starts with a main() function. The fn keyword declares functions, and statements end with semicolons. Rust is statically typed but uses type inference, so you rarely need to write types explicitly. The Rust compiler (rustc) is your best friend - it catches errors before runtime!

Key Points:

  • fn main() is the entry point of every Rust program
  • println! is a macro that prints to the console (note the !)
  • let declares variables (immutable by default)
  • let mut declares mutable variables
  • Rust uses semicolons (;) to end statements
  • Functions are defined with fn function_name(parameters) -> return_type
  • Comments start with //
  • Rust has strong type inference but explicit type annotations are recommended for clarity

Rust Foundation Explained

Rust is designed for performance and safety, particularly safe concurrency. Unlike C or C++, Rust is memory-safe without using garbage collection. It achieves this through a unique ownership system that the compiler checks at compile time. This means no null pointer dereferencing, no data races, and no segfaults at runtime!

The main() function is where every executable Rust program starts execution. Rust programs are compiled to native code, which makes them extremely fast. The Rust compiler is famously strict - it won't let you compile code with potential safety issues, which saves countless hours of debugging runtime errors.

Zero-Cost Abstractions

Rust provides high-level programming constructs (like iterators, traits, and pattern matching) that compile down to code as efficient as hand-written C, with no runtime overhead.

Memory Safety

Rust's ownership system ensures memory safety without garbage collection. The compiler tracks variable lifetimes and prevents dangling pointers, buffer overflows, and use-after-free errors.

Performance

Rust competes with C++ in performance while providing modern language features and safety guarantees. It's used in performance-critical systems from operating systems to game engines.

Real-World Application

Rust is used by major companies like Microsoft, Google, Amazon, and Facebook for critical infrastructure. Firefox uses Rust in its browser engine (Servo). Discord uses Rust for real-time messaging. Cloudflare uses Rust for edge computing. Dropbox uses Rust for its sync engine. These companies choose Rust because it eliminates entire classes of bugs while delivering C++-level performance.

Module 2: Variables and Data Types

Rust is a statically typed language, which means the type of every variable must be known at compile time. However, Rust has excellent type inference, so you often don't need to write type annotations. Rust's type system prevents many common bugs that occur in other languages.

Rust has several primitive data types: integers (i8, i16, i32, i64, i128, isize), unsigned integers (u8, u16, etc.), floats (f32, f64), booleans (bool), and characters (char). Rust also has compound types like tuples and arrays.

Rust's Type System Explained

Rust's type system is designed to catch bugs at compile time rather than runtime. The compiler uses type information to ensure memory safety, prevent data races, and eliminate entire classes of errors. Rust's type system is expressive yet practical - it supports generics, traits, and algebraic data types while maintaining zero-cost abstractions.

The distinction between signed and unsigned integers is crucial in systems programming. Choosing the right integer type affects both performance and correctness. For example, usize is the pointer-sized integer type used for indexing collections, while u8 is perfect for working with bytes. Rust's strict type checking prevents common bugs like integer overflow in debug builds.

Memory Layout

Rust's types have predictable memory layouts with no hidden overhead. Structs and enums are stored inline, and the compiler optimizes padding and alignment for performance.

Numeric Safety

Rust prevents implicit type conversions that could lose precision. You must use explicit as casts, making conversions visible and intentional in the code.

Unicode Support

Rust's char type represents a Unicode scalar value, not a byte. This prevents common encoding bugs and ensures proper internationalization support from day one.

Systems Programming Applications

In embedded systems, choosing the right integer type is critical for memory-constrained environments. Network protocols use specific byte sizes that map directly to Rust's integer types. Game engines use Rust's float types for physics calculations. Database systems use Rust's type system to ensure data integrity. Financial systems rely on Rust's integer overflow checks. Operating systems use Rust's pointer-sized types (usize/isize) for memory addressing. In each case, Rust's precise type system provides both safety and performance.

OWNERSHIP & BORROWING

Learn Rust's unique memory management system

Module 4: Ownership and Borrowing

Ownership is Rust's most unique feature that enables memory safety without garbage collection. The ownership system is governed by three rules: 1) Each value has an owner, 2) There can only be one owner at a time, 3) When the owner goes out of scope, the value is dropped.

Borrowing allows you to reference data without taking ownership. References are either mutable (&mut T) or immutable (&T). The borrow checker enforces rules at compile time: you can have either one mutable reference or multiple immutable references, but not both at the same time. This prevents data races!

Memory Safety Without Garbage Collection

Rust's ownership system is its most innovative feature, enabling memory safety without runtime overhead. By tracking ownership at compile time, Rust eliminates entire classes of bugs: use-after-free, double-free, data races, and memory leaks (aside from reference cycles). The system is based on three core concepts: ownership (who owns memory), borrowing (temporary access), and lifetimes (how long references are valid).

The borrow checker is Rust's compile-time guardian that enforces these rules. It analyzes your code to ensure references don't outlive the data they point to, and that mutable access is properly synchronized. While initially challenging for programmers from garbage-collected languages, the ownership system becomes intuitive with practice and enables fearless concurrency and optimal performance.

Zero-Cost Abstraction

Ownership tracking happens at compile time with no runtime overhead. The compiled code is as efficient as hand-written C with manual memory management.

Fearless Concurrency

Ownership rules prevent data races at compile time. You can share data between threads without locks, or the compiler will tell you exactly why it's unsafe.

Bug Prevention

Rust catches memory safety bugs that would cause crashes, security vulnerabilities, or undefined behavior in other languages, all before the program runs.

Systems & Enterprise Applications

Web browsers (like Firefox's Servo engine) use Rust's ownership for safe parallel DOM manipulation. Operating systems (like Redox OS) use it for kernel memory safety. Database systems use it for transaction isolation. Game engines use it for safe resource management. Web servers use it for connection pooling. Cryptography libraries use it to prevent timing attacks. In each case, Rust's ownership system provides the safety guarantees needed for critical software while maintaining the performance required for systems programming.

Module 5: Structs and Enums

Structs and enums are Rust's way of creating custom data types. Structs are similar to classes in other languages (but without inheritance), while enums allow you to define a type that can be one of several variants. Together with pattern matching, they form a powerful system for data modeling.

Structs group related data together. They can have methods defined in impl blocks. Enums are particularly powerful in Rust because each variant can have different associated data. The Option and Result enums are built into Rust's standard library and used extensively for error handling and representing optional values.

Algebraic Data Types in Rust

Rust's structs and enums form a system of algebraic data types that provide both safety and expressiveness. Structs are product types (combinations of fields), while enums are sum types (alternatives between variants). This mathematical foundation enables the compiler to reason about your code and catch bugs at compile time. Pattern matching makes working with these types both safe and elegant.

The Option and Result enums are fundamental to Rust's philosophy of explicit error handling. Unlike languages with null pointers or exceptions, Rust requires you to handle the possibility of absence or failure explicitly. This eliminates null pointer exceptions and makes error paths visible in the code. The compiler ensures you handle both Some/Ok and None/Err cases when using match.

Zero-Cost Enums

Rust enums are stored efficiently, using only enough space for the largest variant plus a discriminant. The compiler eliminates the discriminant when possible through niche optimization.

Pattern Matching

match expressions provide exhaustive pattern matching that works with structs, enums, tuples, and literals. The compiler ensures all cases are handled.

Error Handling

The Result type makes error handling explicit and composable. Error propagation with ? operator makes error handling concise while maintaining safety.

Data Modeling & APIs

Web APIs use structs to represent request/response bodies and enums to represent different endpoint types. Database ORMs use structs for table rows. GUI frameworks use enums for events. Game engines use structs for entities and enums for states. Network protocols use enums for message types. Compilers use enums for AST nodes. Configuration systems use structs for settings. In each case, Rust's type system provides a precise, safe way to model domain concepts that catches invalid states at compile time.

Module 6: Error Handling in Rust

Rust doesn't have exceptions. Instead, it uses the Result<T, E> type for recoverable errors and the panic! macro for unrecoverable errors. This explicit error handling makes error paths visible in your code and forces you to think about error cases.

The ? operator provides ergonomic error propagation, making it easy to return errors from functions without verbose match statements. Rust's error handling is composable - you can easily convert between different error types, and the ecosystem provides crates like anyhow and thiserror for application and library error handling respectively.

Explicit, Safe Error Handling

Rust's error handling philosophy is based on explicitness and safety. Unlike languages with exceptions, where error paths are invisible in function signatures, Rust makes errors part of the type system. The Result type forces you to acknowledge that operations can fail. This eliminates surprises and makes code more robust. The compiler ensures you handle errors, either explicitly or by propagating them.

The ? operator is Rust's elegant solution to error propagation. It automatically converts and returns errors, reducing boilerplate while maintaining safety. Combined with custom error types and the From trait, you can create sophisticated error handling that's both ergonomic and type-safe. The ecosystem provides crates like anyhow for quick application development and thiserror for robust library error types.

Error Visibility

Errors are part of function signatures, making error paths explicit. You can see at a glance which functions can fail and what errors they return.

Error Propagation

The ? operator makes error propagation concise while maintaining type safety. It automatically converts between compatible error types.

Error Composition

Rust's error types compose naturally. You can create error hierarchies, convert between error types, and handle errors at the appropriate level.

Production Systems & APIs

Web servers use Result to handle HTTP errors gracefully. Database drivers use custom error types for SQL errors. File parsers use error propagation to handle malformed data. Network clients use error types for connection failures. Configuration systems use Result for validation errors. Cryptography libraries use precise error types for security failures. In each case, Rust's error handling provides the safety guarantees needed for reliable systems while making error paths explicit and manageable.

Efficient, Safe Collections

Rust's collections are designed with both safety and performance in mind. They provide bounds checking to prevent buffer overflows, yet compile to efficient code comparable to C++ STL containers. The ownership system ensures that collections properly manage their memory, preventing use-after-free and memory leaks. Collections work seamlessly with Rust's iterator system, enabling functional-style operations like map, filter, and collect.

Each collection has specific performance characteristics that make it suitable for different use cases. Vec provides fast random access but slow insertions at the beginning. HashMap provides average O(1) operations but requires hashable keys. String guarantees valid UTF-8, preventing encoding bugs. The standard library provides these carefully designed abstractions so you can focus on solving problems rather than implementing data structures.

Performance Guarantees

Rust's collections have documented time and space complexity guarantees. The implementations are optimized and tested, providing predictable performance.

Memory Safety

Collections manage their own memory, preventing buffer overflows, use-after-free, and other memory errors. Bounds checking catches out-of-bounds access.

All standard collections work with Rust's iterator system, enabling functional programming patterns that are both expressive and efficient.

Data Processing & Systems

Compilers use Vec for AST nodes and HashMap for symbol tables. Network stacks use VecDeque for packet buffers. Configuration systems use collections for hierarchical data. In each case, Rust's collections provide the right balance of safety, performance, and ergonomics for systems programming.

Module 8: Concurrency in Rust

Rust's ownership system enables "fearless concurrency" - writing concurrent code that's free of data races at compile time. Rust provides several concurrency primitives: threads for parallel execution, channels for message passing, and synchronization types like Mutex and Arc for shared state.

Unlike many languages, Rust prevents data races at compile time. The type system ensures that data is either uniquely owned (and mutable) or shared (and immutable). For shared mutable state, Rust provides synchronization primitives that enforce safe access patterns. This eliminates entire classes of concurrency bugs.

Fearless Concurrency

Rust's concurrency model is built on the foundation of its ownership system. The compiler statically prevents data races - concurrent access to mutable data - which are a common source of bugs in concurrent programs. This is achieved through the Send and Sync traits, which are automatically implemented for types that can safely be transferred or shared between threads. If your code compiles, it's free of data races.

Rust provides multiple concurrency paradigms: message passing with channels (inspired by Go), shared state with mutexes (traditional but safe), and lock-free programming with atomics. Each has its trade-offs. Channels avoid shared state entirely, making reasoning about concurrency easier. Mutexes are familiar but require careful use to avoid deadlocks. Atomics provide the highest performance but are limited to simple operations.

Data Race Prevention

The compiler ensures that data is either uniquely owned (and mutable) or shared (and immutable). This eliminates data races at compile time, a unique feature among systems programming languages.

Message Passing

Channels provide a safe way for threads to communicate without sharing memory. This Erlang/Go-inspired model often leads to simpler, more maintainable concurrent code.

Zero-Cost Abstractions

Rust's concurrency primitives compile to efficient machine code. Mutex and Arc have minimal overhead, and channels are highly optimized.

High-Performance Systems

Web servers like Actix and Rocket use Rust's concurrency for handling thousands of simultaneous connections. Databases like Sled use it for concurrent access. Game engines use it for parallel rendering. Blockchain nodes use it for peer-to-peer networking. Operating systems use it for kernel scheduling. Video processing pipelines use it for frame parallelization. In each case, Rust's concurrency guarantees enable building high-performance, reliable systems without the typical concurrency bugs.
